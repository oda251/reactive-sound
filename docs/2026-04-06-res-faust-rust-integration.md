# Faust と Rust の統合方法調査

## 1. faust2rust: Faust コードを Rust に直接コンパイル

Faust コンパイラは `-lang rust` オプションで Rust コードを直接生成できる（[Faust Compiler Options](https://faustdoc.grame.fr/manual/options/)）。

**使い方:**

```bash
# Rust コードを生成
faust -lang rust -a <architecture_file> mydsp.dsp -o mydsp.rs

# JACK + Rust バイナリを直接ビルド
faust2jackrust mydsp.dsp

# PortAudio + Rust バイナリを直接ビルド
faust2portaudiorust mydsp.dsp
```

**Rust 専用コンパイラオプション**（[Faust Compiler Options](https://faustdoc.grame.fr/manual/options/)）:

| フラグ | 説明 |
|--------|------|
| `-noreprc` | DSP 構造体の C ABI レイアウト強制を無効化 |
| `-rnt` | `FaustDsp` トレイト実装を生成しない |
| `-rnlm` | libm への FFI 呼び出しを生成しない |

**制約:**
- アーキテクチャファイルが C++ ほど充実していない（`faust2jackrust` と `faust2portaudiorust` のみ）（[faust2[...] Tools](https://faustdoc.grame.fr/manual/tools/)）
- 生成コードは可読性が低く、手動での修正・拡張が難しい（[Glicol Issue #25](https://github.com/chaosprint/glicol/issues/25) での議論）

## 2. Rust バインディング / クレート

### rust-faust (Frando/rust-faust)

ビルド時に Faust DSP を Rust モジュールにコンパイルする仕組み（[GitHub: rust-faust](https://github.com/Frando/rust-faust)）。

| 項目 | 状態 |
|------|------|
| 最終コミット | 2025年1月26日 |
| Stars | 約80 |
| ライセンス | Apache-2.0 / MIT デュアル |
| ステータス | **WIP（開発中）** |

**構成クレート:**
- `faust-build`: `build.rs` で Faust DSP ファイルを Rust モジュールにコンパイル
- `faust-types`: 生成コードが使う型・トレイト定義
- `faust-state`: パラメータ管理の抽象化
- `faust-macro`: Rust ファイル内に Faust DSP コードを直接記述可能

**評価:** 最も実用的な静的コンパイルアプローチ。ただし WIP であり、ドキュメントが乏しい。

### faust_jit (nih-faust-jit 内)

libfaust の LLVM JIT を Rust から利用するクレート（[GitHub: nih-faust-jit](https://github.com/YPares/nih-faust-jit)）。

- `SingletonDsp` 型で DSP スクリプトの読み込み・オーディオ処理・パラメータ抽出を提供
- ステレオ制限はプラグイン側のみ、クレート自体は任意チャンネル対応
- Nix flake によるビルドサポートあり
- コミット数 101、リリース版なし

### faustus (tonal-glyph/faustus)

Faust のクリエイティブコーディングツールキット（Rust 製）（[GitHub: faustus](https://github.com/tonal-glyph/faustus)）。詳細は未検証。

## 3. libfaust (JIT): 動的 DSP コード変更

libfaust は Faust コンパイラを C++ ライブラリとして組み込み、LLVM JIT で実行時にネイティブコードを生成する仕組み（[Embedding the Compiler](https://faustdoc.grame.fr/manual/embedding/)）。

**Rust からの利用方法:**
- `faust_jit` クレート（nih-faust-jit 由来）が libfaust の C API を FFI 経由でラップしている（[GitHub: nih-faust-jit](https://github.com/YPares/nih-faust-jit)）
- libfaust 自体は C++ ライブラリだが、C API (`libfaust-c`) が提供されており FFI 可能（[Embedding the Compiler](https://faustdoc.grame.fr/manual/embedding/)）

**利点:**
- 実行時に DSP コードを変更・リロード可能（ホットリロード）
- DAW 上で Faust スクリプトを編集しリアルタイム反映するユースケースに適する

**制約:**
- libfaust + LLVM のリンクが必要で、ビルド環境の構築が複雑
- バイナリサイズが大きくなる（LLVM ランタイム同梱）
- `faust_jit` クレートは crates.io 未公開の可能性が高い（未検証）

## 4. Faust → WASM → Rust

Faust は WASM バックエンドを持ち、`faust2wasm` スクリプトで WebAssembly を直接生成できる（[FAUST Domain Specific Audio DSP Language Compiled to WebAssembly](https://dl.acm.org/doi/fullHtml/10.1145/3184558.3185970)）。

**経路:**
```
Faust DSP → faust -lang wasm → .wasm バイナリ → Rust (wasmtime/wasmer) で実行
```

**利点:**
- サンドボックス実行で安全
- libfaust 不要（コンパイル済み WASM を配布するだけ）
- クロスプラットフォーム

**制約:**
- WASM ランタイムのオーバーヘッド（リアルタイムオーディオでは問題になりうる）
- Faust WASM 出力は主に Web Audio API 向けに設計されており、ネイティブ Rust から呼ぶ場合は独自のグルーコードが必要（未検証）
- Rust の WASM ランタイム（wasmtime, wasmer）との統合事例は確認できなかった（未検証）

## 5. Glicol + Faust の連携事例

Glicol（Rust 製のグラフ指向ライブコーディング言語）で Faust ノードの追加が提案されたが、**採用されなかった**（[GitHub Issue #25](https://github.com/chaosprint/glicol/issues/25)）。

**不採用の理由:**
- **ライセンス:** Glicol は MIT、Faust のライブラリとの互換性に懸念
- **可読性:** Faust 生成コードはボイラープレートが多く、初心者に不親切
- **メンテナンス:** 手書き Rust の方がデバッグしやすい

**結論:** Faust → Rust 生成コードを別プロジェクトに組み込む実用事例は、nih-faust-jit（VST3/CLAP プラグイン）以外にはほとんど見当たらなかった。

## 6. ライセンス

### Faust コンパイラ本体

**LGPL**（GPL ではない）ライセンスで提供されている（[Faust FAQ](https://faustdoc.grame.fr/manual/faq/)）。

### 生成コードのライセンス

> "You can freely use Faust to develop proprietary software. The LGPL license of the compiler _doesn't_ apply to the code generated by the compiler."
> -- [Faust FAQ](https://faustdoc.grame.fr/manual/faq/)

- 生成コードのライセンスは**入力ファイル（DSP コード + 使用ライブラリ）のライセンスに依存**する
- Faust 標準ライブラリとアーキテクチャファイルは、未改変であれば**プロプライエタリ利用と互換**（[Faust FAQ](https://faustdoc.grame.fr/manual/faq/)）
- つまり、自分で書いた Faust DSP + 標準ライブラリの組み合わせであれば、**商用・クローズドソースアプリに含めて問題ない**

### 注意点

- サードパーティの Faust ライブラリを使う場合は、そのライブラリ個別のライセンスを確認する必要がある
- libfaust 自体を動的リンクで使う場合は LGPL の条件（動的リンク or ソース提供）に従う必要がある（未検証）

## まとめ: 最も実用的な統合方法

| 方法 | 実用度 | 動的変更 | ビルド複雑度 | 備考 |
|------|--------|----------|-------------|------|
| `faust -lang rust` + 手動統合 | 中 | 不可 | 低 | 生コマンドで生成、自分でビルドに組み込む |
| **rust-faust (faust-build)** | **高** | 不可 | **低** | `build.rs` で自動化。最も手軽 |
| faust_jit (libfaust JIT) | 中 | **可能** | 高 | ホットリロード可。LLVM 依存 |
| Faust → WASM → Rust | 低 | 可能 | 中 | オーバーヘッド大。事例少 |

**推奨:**
- **静的 DSP を組み込む場合** → rust-faust (`faust-build`) が最も実用的。`build.rs` に Faust コンパイルを統合でき、生成コードは `FaustDsp` トレイトを実装する
- **実行時に DSP を変更したい場合** → `faust_jit` クレート（libfaust LLVM JIT）が唯一の選択肢だが、ビルド環境構築のハードルが高い
